SELinux 那点事儿

发表于 2019-01-27 | 分类于 Security

1. SELinux的概述

Android安全模型是基于应用沙盒的，即每个应用都在自己的沙盒中运行，拥有独一无二的UID。为了更好保护、限制系统服务，控制对应用数据及系统日志的访问，降低恶意软件的影响，以进一步提升系统安全性，Google将Linux系统的强制访问机制(MAC) - SELinux(Security Enhanced Linux)引入到了Android系统中。

一旦启用SELinux，任何未经允许的行为都会被拒绝，且dmeg和logcat中会打印出这些行为信息。SELinux主要包含2种模式(1)Permissive mode，在该模式下权限拒绝行为会被记录下来，但不会被强制执行；(2)Enforceing mode，在该模式权限拒绝行为会被记录下来并被强制执行。

SELinux主要依靠label来匹配action和policy，label用于决定允许的事项，socket、file以及process都有label。其中policy的形式是allow domains types:classes permissions，domain是一个或一组process的label(域类型)，type是一个或一组对象的label，class是要访问的对象的类型，permission是要执行的操作。

除了在rule中逐个列出domain或type外，还可以通过attribute引用一组domain或type。attribute是一组domain或type的名称，每个domain或type都可以与任意数量的attribute关联。当编写的rule指定了某个attribute名称时，该名称会自动扩展为与该attribute关联的所有domain或type。

SELinux政策的avc规则形式：
RULE_VARIANT SOURCE_TYPES TARGET_TYPES : CLASSES PERMISSIONS
表示当任何source_types标签的主体尝试对包含任何classes类且带有target_types标签的对象执行与任何permissions对应的操作时，应该发生什么情况。

Keymaster 的 SPL 版本绑定

发表于 2019-01-12 | 分类于 Security

在Keymaster 1.0中，所有Keymaster密钥都会绑定至信任根或Verified Boot密钥。在Keymaster 2.0和3.0中，所有Keymaster密钥也绑定至OS及系统image的Security Patch Level(SPL)；可确保发现旧软件漏洞的人无法将手机系统还原至包含漏洞的软件版本，也无法使用在新软件中所创建的密钥；当系统升级时，密钥会随着系统一起“升级”。在Keymaster 4.0中，为适配Treble模块化结构，boot,system,vendor等多个系统分区均包含SPL信息，另外支持AVB的系统还会将这些SPL及OS版本信息存入vbmeta中，以便bootloader可以将这些内容传给Keymaster。

在实际系统开发中，经常会遇到合入新的Google Security Patch造成CTS/VTS等认证测试失败，而版本发布计划又不能推迟，最后不得不采用临时回退安全补丁方式的事情发生。这样就很容易收到测试人员反馈：半擦升级/OTA升级进入输入密码界面，分析日志不难发现，都是上述提到的SPL绑定Keymaster惹的祸。下面就来跟一下Log打印流程：

Userdata 分区加密方式

发表于 2019-01-05 | 分类于 Security

Android系统主要支持FDE/FBE两种加密方式。
Android P及更高版本支持元数据加密。该密钥受keymaster保护，而keymaster受verified boot保护。使用元数据加密时，启动出现的单个密钥会加密未通过文件级加密进行加密的任何内容。

一、 Full-disk encryption(FDE)

1.1 概述

Android L及更高版本支持全盘加密。全盘加密是使用密钥（由用户设备密码保护）对设备上用户数据进行编码的过程。设备加密后，所有由用户创建的数据在存入磁盘前都会字典加密，且所有读取操作都会在将数据返回给调用进程之前自动解密数据。用户必须提供credentials才能访问磁盘。

全盘加密采用128位AES加密算法，支持‘默认’，‘PIN码’，‘密码’，‘Pattern图案解锁’四种加密状态。首次启动，会先随机生成128位主密钥，然后使用默认密码’default_password’（cryptfs.cpp）和存储的salt对其进行hash处理，接着由TEE生成hash签名以加密该主密钥。（16kb）

快速加密方式：只对userdata分区已使用的分块进行加密，避免首次开机耗时过长。目前仅ext4和f2fs类型文件系统支持该快速加密方式。

加密过程由init/vold进程管理。init负责调用vold，然后vold来设置相关属性以触发init中事件。系统会使用vdc的cryptfs命令来调用vold中的加密功能。主要命令有：checkpw、restart、enablecrypto、changepw、cryptocomplete、verifypw、setfield、getfield、mountdefaultencrypted、getpwtype、getpw、clearpw等。

无论加解密还是清除data分区过程，data分区都不能处于挂载状态。所以为了确保framework启动，data分区就需要挂载一个临时文件系统，使得android可提示输入密码以清除数据此类建议。

在实际工作中可能会遇到一些加解密错误，比如密码匹配无法解密，用户输错密码此次达到30次。若这些错误未解决，则会提示用户清除数据恢复出厂设置。

AB 系统升级概述

发表于 2018-11-24 | 分类于 OTA

一、A/B系统更新的特点

 所谓A/B系统，即有A和B两套系统，一套系统分区，另一套备份分区。这两套系统出厂时一样，此后可能不一样。即一个新版本，另一个旧版本，旧版本升级至新版本，不断更新切换。

 A/B系统更新又称之为无缝更新(Seamless updates),具有以下几个特点：

 <1>OTA更新在系统后台运行，该过程中用户仍可以正常使用设备。待更新完成后，需重启一次方可进入新系统。

 <2>OTA更新失败后(OTA更新无法应用或应用后无法启动)，设备可重启回滚到旧分区继续使用，并重新尝试更新升级。

 <3>可采用流式升级方式，即无需/data或/cache分区留出足够空间用于存储下载的升级包，也可进行OTA升级。

 <4>确保在OTA更新期间磁盘上保留一套可以正常启动使用的系统，减少刷机变砖的可能性，减轻售后服务工作量。
