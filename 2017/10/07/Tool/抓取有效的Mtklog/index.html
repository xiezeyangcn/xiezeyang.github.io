抓取有效的 MTKLog

发表于 2017-10-07 | 分类于 Tool

一、MTKLogger抓取流程：

1.同步时间：手机系统与当前实际一致。
2.清除 log：如果 MTKLogger 已经启动，先停止并等待完成清除旧的 log 文件。
3.启动 MTKLogger：测试复现问题， 注意复现步骤，留心规律。
4.异常发生：记录问题次数和对应时间点，部分问题也需记录一下频率（如monkey 、MTBF等长时间测试的问题）
5.停止MTKLogger：停止 Log ，并等待停止完成，然后导出 log 文件。

二、实际应用：

1.一般逻辑性 逻辑性 /功能缺陷问题：

<1>参照上述MTKLogger 抓取流程，提供 log 。

<2>异常行为界面截图标注。

<3>其它对比机器行为如何，正确也需截图标注出的。

2.应用程序 ANR 、FC 问题：

<1>参照上述使用MTKLogger 抓取流程，提供 log 。

<2>异常界面截图或者录制视频，如是截屏需要保证实时性，最好是录制视频。

3.两台台设备的交互类异常（如 DUT A和 DUT B通话 /短彩信等）：

<1>MO/MT端都需参照上述MTKLogger 抓取流程，提供 log 。

<2>如该异常为上述提到的实际应用 1，则提供 1所需的其他文件

4.死机重启类异常：

<1>重启前 \后的 MTKlogger需要提供

<2>提供问题视频

<3>以下概率性问题，请不要破坏现场，把机器提供给研发抓取 log和分析：
不停的重启，死机后不能开机，开机过程中卡死

5.Monkey测试注意问题（测试前）：

<1>需要设置 ro.monkey 值为true。
如为 eng 版本，则可通过在命令行中输入:adb shell setprop ro.monkey true ，来设置该属性。（如为 useruser 版本，则让研发提供设置该属性的工程指令，目前尚未合入相关指令）

<2>使用充电器，而非电脑 usb 供电测试。

<3>一般只需开启 mobile log中的 main_log和 kernel_log 即可，其他可关闭

<4>参照上述MTKLogger 抓取流程，提供 log

三、MTKLogger自动清理机制：

1.MobileLog自动清理机制：
考虑到Log占用空间因素， MobileLog会设定 Log 大小的限制。每种类型大小的限制。每种类型 log （main 、radio 、events、kernel 、bt） 最多存在 5个文件，每个文件都会有相应的 size 限制，当 log 文件写满之后， MobileLog 会进行自动清理：
删除 xxx_log.4， xxx_log.3–>xxx_log.4 >，xxx_log.2–>xxx_log.3 xxx_log.1 –>xxx_log.2 ，并创建一 个新的文件 xxx_log，其中 “–>” 表示重命名。文件名中 “.”后面的数字越大，log 文件越旧，并且被删除件中的 log 信息就会丢失。

2.设置方法：
MTKLogger –>Settings –>MobileLog>–>Limit Current Log Size(MB)/Total Log Size( MB )，这里有两项设置，前者指的是本次抓取MobileLog大小， 开始录的时候就会新建一个APlog_XXX 的文件夹， XXX 就是开始录 log 的时间， 后者是指所有的MobileLog大小。（一般将前者设置为 1024M ，这样mobilelog文件下每种类型的 log 文件大小最约为 1024/20=50M，如果文件太大也不方便研发打开分析，后者设置为 2048M）

<1>检查非当前正在录制 log 的目录
如果包含 Modem Exception，对于 L1/PS/DSP 类型的 log ，仅保留最早、次新、最新的 dmp/muxraw文件，如果不包含 Modem Exception，则删除整个目录

<2>检查当前正在录制 log 的目录，对于 L1/PS/DSP类型的log，仅保留最早、 次新、最新的 log 文件

<3>如果上述<1>、<2>步骤中没有文件被删除，则选择最旧的目录设置方法： MTKLogger–>S etting –>MobileLog–>Limit Log Size(MB)

<4>针对比如场测、monkey长时间测试，建议在MTKLogger工具中将存储位置由Phone改为SD卡。
